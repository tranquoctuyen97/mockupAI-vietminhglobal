## 2. Root cause analysis chi tiết

### Bug #1+#2 — Performance hang

**Chain reaction**:

```
User click "Chọn hết" 
  → selectAllColors() 
  → updateDraft({enabledColorIds}) 
  → optimistic update + debounce 1s timer
  
After 1s:
  → saveDraftImmediately() 
  → PATCH /api/wizard/drafts/:id
     ├─ updateDraft() in DB (1 call)
     ├─ findUnique select mockupsStale (1 call)
     ├─ hasActiveMockupJob() (1 call)
     ├─ regenerateMockupsForDraft() — stub, just resetStaleFlag (1 call)
     └─ Returns updated draft
  → Frontend set({ draft: {...updated} })
  → Effect [draft?.id, draftEnabledColorKey] runs
  → Effect [draft?.id, draftPlacementOverrideKey] runs
  → Effect [draft?.id, draft?.storeId, retryNonce] runs → REFETCH 3 APIs!
     ├─ /api/stores/:id/template
     ├─ /api/stores/:id/colors
     └─ /api/stores/:id/preset-status
  → setLoading(true) hiện spinner!
```

**Vấn đề**: Mỗi PATCH save trigger 3 fetches lại template/colors/preset-status. Effect dependency `draft?.storeId` không thay đổi nhưng `draft?.id` cùng instance → React phải so deepEqual trên primitive (OK) nhưng draft object reference mới → React re-runs effect.

Wait, `draft?.id` là string nên === stable. Effect không re-run. Nhưng `retryNonce` không thay đổi unless user explicitly retry.

Hmm. Re-reading: line 137 `}, [draft?.id, draft?.storeId, draftId, retryNonce]);` — chỉ trigger khi 1 trong 4 thay đổi. `draft?.id` và `draft?.storeId` không thay đổi sau save (server returns same id/storeId). `draftId` from URL. `retryNonce` chỉ tăng khi click "Thử lại". OK so the 3-API-fetch effect only runs once on mount.

So performance issue is more subtle. Let me re-think...

**Probable culprit**: 
- `draftPlacementOverrideKey = JSON.stringify(draft?.placementOverride ?? null)` — tính trên mỗi render. Nếu placementOverride là object lớn, JSON.stringify chậm.
- 8 useEffects, mỗi cái có deps. React re-runs các effects sau mỗi state change.
- Polling effect at line 160 polls every 2s when mockupJobId set — even after job completed.

**Real bottleneck**: dev server `next dev` mode rebuild + multiple parallel API calls. In production mode chắc nhanh hơn. Nhưng vẫn cần tối ưu.

#### Fix 1.2 — Add timeout + AbortController cho mọi fetch nặng

**File**: `app/src/app/(authed)/wizard/[draftId]/step-4/page.tsx`

```tsx
async function handleGenerateAI() {
  setState("ai-loading");
  setAiError(null);

  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort(), 60_000); // 60s timeout

  try {
    const res = await fetch(`/api/wizard/drafts/${draftId}/generate-content`, {
      method: "POST",
      signal: controller.signal,
    });
    clearTimeout(timeoutId);
    // ... existing logic
  } catch (err) {
    clearTimeout(timeoutId);
    if (err.name === "AbortError") {
      setAiError({
        code: "timeout",
        userMessage: "AI quá tải. Vui lòng thử lại sau hoặc viết tay.",
        retryable: true,
        severity: "error",
      });
    } else {
      setAiError(parseAIError(new Error("Network error")));
    }
    setState("ai-failed");
  }
}
```

Cũng add timeout server-side:

**File**: `app/src/app/api/wizard/drafts/[id]/generate-content/route.ts`

```typescript
const GENERATE_TIMEOUT_MS = 45_000;

const result = await Promise.race([
  withRetry(() => generator.generate(input), { maxAttempts: 2 }),  // giảm 3 → 2
  new Promise((_, reject) =>
    setTimeout(() => reject(new Error("AI provider timeout")), GENERATE_TIMEOUT_MS),
  ),
]);
```

Apply same pattern cho:
- `/api/mockup-jobs` POST (60s)
- `/api/wizard/drafts/:id/publish` POST (90s)
- Mọi fetch trong client wizard pages

**Test**: Click "Tạo nội dung AI" với network throttle (Slow 3G). Timeout 60s → error message clear.

### Sprint 4 — Performance + dev experience (P2, ~3 giờ)

#### Fix 4.1 — Reduce useEffect count step-3

Combine effects bằng `useReducer` hoặc state machine. Hiện tại 8 effects:
1. Load draft if not hydrated
2. Sync selectedColorIds from draft
3. Sync placementOverride from draft
4. Fetch template/colors/preset
5. Pick latest mockupJobId
6. Fetch design preview URL
7. Poll mockup job status
8. Set initial livePreviewView

→ Consolidate xuống 3-4 effects.

#### Fix 4.2 — Memoize heavy computations

```tsx
const draftPlacementOverrideKey = useMemo(
  () => JSON.stringify(draft?.placementOverride ?? null),
  [draft?.placementOverride],
);
```

#### Fix 4.3 — Lazy-load LivePreview

```tsx
const LivePreview = dynamic(() => import("@/components/mockup/LivePreview").then(m => ({ default: m.LivePreview })), {
  ssr: false,
  loading: () => <Skeleton height={420} />,
});
```

→ Initial JS bundle nhỏ hơn → page load nhanh hơn.

#### Fix 4.4 — Stop polling khi job completed

Hiện tại polling stop khi status !== running/pending, nhưng `mockupJobId` vẫn set → effect re-trigger không cần thiết khi component re-render.

```tsx
useEffect(() => {
  if (!mockupJobId) return;
  let stopped = false;

  const pollJob = async () => {
    if (stopped) return;
    // ... existing logic
    if (job.status === "running" || job.status === "pending") {
      timeoutId = setTimeout(pollJob, 2000);
    } else {
      stopped = true;  // ← explicit stop
    }
  };

  pollJob();
  return () => { stopped = true; if (timeoutId) clearTimeout(timeoutId); };
}, [mockupJobId, draftId]);
```
